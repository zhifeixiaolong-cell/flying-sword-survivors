## §1 设计哲学

**M3 不是"敌人系统",是"M2 飞剑设计的真实压力验证场"。**

M2 飞剑系统是在真空中提出的——单剑容量是张力源、池空那一刻是"张力时刻"、双程轨迹的物理涌现是核心体验。但 M2 自己玩的时候**没有敌人**,这些张力假设没有压力来源,只是设计师在空场上调参时的体感。

M3 第一次给这些假设接上真实压力。所以 M3 的设计目标不是"做出敌人",而是**验证 M2 的设计假设是否在真实压力下成立**:

- 池空那一刻,有真实威胁逼近时,玩家是否真的感受到"张力"而不是"麻木"?
- 单剑容量瓶颈,在敌人持续涌入时,是否真的逼出"取舍 / 节奏 / 专注"的体验?
- 飞剑双程轨迹的视觉爽感,在战斗中而不是空场时,是否还能被感知?

M3 选择 **survivor 哲学**——持续涌入、无 You Win、撑多久就多久——而非关卡制(打完一关进下一关)。理由是 survivor 模式最能放大"持续紧绷"的体验,而这正是验证 M2 单剑张力假设的最佳压力场。

**M3 同时是首个独立可演示里程碑**。M0-M2 都是无失败状态的演示——可以打,但不会输。M3 第一次让游戏有"开始-失败-重来"的完整循环,即使失败状态是 placeholder。这让 M3 ship 之后可以作为完整片段给人看,不需要语言解释"想象一下加上敌人会怎样"。

## §2 边界与不变量

### M3 守护的不变量

M3 不破坏 M2 已建立的三大不变量(详见 `docs/design-invariants.md`):

- **INV-01** 飞剑速度恒 > 角色速度 — M3 不调整 `PLAYER_SPEED` 或 `SWORD_MIN_SPEED`,M2 余量 210 px/s 保持
- **INV-02** 飞剑数量 = 资源池容量 — M3 不引入"双剑临时态"等机制
- **INV-03** 机制成长 = 修为成长 — M3 不引入"升级 / 解锁"等关卡进度元素

### M3 做的最小集

- 2 种敌人类(脆 + 耐)
- 持续生成系统(EnemySpawner)
- 阶段递增控制(PhaseController)
- 触墙判定(敌人 y 到达城墙顶 y → game over)
- Placeholder game over overlay(冻结场景 + 文字 + restart 提示)
- Restart 流程(按 R → 场景重置)
- BootScene 引入(资源预加载架构,M3 任务可能为空,M5 接美术资产时用)

### M3 不做的事

明确划在 M3 范围外,留待后续里程碑:

- **血量系统 / 城墙血量条** — M4
- **正式 game over UI / 统计 / 动画** — M4
- **关卡进度 / 存档 / 难度选择 / 开始菜单** — M6+
- **敌人速度差异** — 留待 M5+ 多剑武器扩展时再考虑
- **第三种敌人 / boss** — 留待 M5+
- **奖励 / 击杀计数 / 经验** — M5+(若引入)
- **音效 / 配乐** — M5+ 美术资产阶段
- **性能优化(敌人对象池 / culling)** — M6+(若需要)

## §3 核心张力定义

### 设计目标 framing

M3 的核心张力 framing 是 **"专心就赢,分神就输"**。这是体验目标,需要翻译成可设计的机制语言。

### 玩家 DPS 物理上限推导

M2 飞剑数值已定(详见 `docs/m2-sword-system-design.md` §8):
- `V_max = 1200 px/s`,`V_min = 360 px/s`,`D_max = 600 px`
- 单剑容量 = 1(INV-02)
- 锋利度 = 0.7,伤害系数 = 0.1,追踪角速度上限 = 180°/s

由此估算单剑完整周期(出鞘 → 衰减 → 转折 → Homing 回程 → 入鞘):

- 去程到 `D_max`:平均速度约 780 px/s(`V_max` 到 `V_min` 线性衰减),走 600 px,约 **0.77s**
- 回程 Homing:角速度 180°/s,距离取决于剑在哪转向,粗估 **0.8 - 1.5s**
- **单剑完整周期 ≈ 1.5 - 2.3s**(估算值,**M3 实施时需实测验证**——M2 设计文档 §8 可能有更准的实测数,本文档定稿时未交叉确认)

这给出了**玩家 DPS 的物理天花板**:每 1.5 - 2.3 秒最多发射一把飞剑。这是不可突破的——因为 INV-02 单剑容量,因为单剑物理周期不能压缩(压缩会破坏 M2 的视觉节奏)。

### "专心就赢,分神就输"的数学条件

把 framing 翻译成可计算的设计目标:

- 如果玩家**完美操作**(每次回收立刻发射、瞄准最优、不浪费空窗):每周期消灭 X 个敌人(X 取决于敌人血量与穿透模型,见 §4)
- 如果玩家**分神**(漏一拍、瞄歪、池空时没注意威胁):每周期 < X 个

设计目标 = **敌人生成速率与"完美操作清场速率"匹配**:
- 完美操作 → 0 漏 → 撑到下一阶段
- 任何失误 → 漏一个 → 触墙 → game over

换句话说,**M3 的难度曲线就是 X 的匹配曲线**——脆敌人 + 耐敌人混合后,玩家"完美操作清场速率"在各阶段是动态变化的,敌人生成速率必须跟着变。这就是 §5 阶段递增系统存在的原因。

### M3 验证什么

M3 实施完成 + 数值调到位之后,以下三个 M2 假设应该被真实压力验证:

1. **"单剑容量是张力源"是否成立?** — 池空时玩家是否感受到紧张?(而不是麻木)
2. **"双程轨迹的物理涌现"在战斗中是否还被感知?** — 还是变成纯功能性的"打怪工具"?
3. **"机制成长 = 修为成长"在 M3 体验中是否合理?** — 玩家 1 分钟后操作是否真有"修为"的提升感?

这三个验证结果会写入未来 M5+ 的多剑武器设计——如果验证失败,M5 多剑要重新考虑设计方向。

## §3.5 游戏生命周期切片(M3 形态)

> **定位说明**:本节描述游戏从启动到失败 / 结束的完整体验流。M3 是首次出现"完整生命周期"的里程碑(此前 M0-M2 都是无失败状态的演示),故在此立节。后续里程碑对生命周期的扩展会在本节迭代更新,而非另起新章。

### M3 生命周期总览

```
[启动 main.ts] 
   ↓
[BootScene]  ← M3 引入,资源预加载(M3 任务可能为空)
   ↓
[MainScene] 
   ↓
[Phase 1: 0-45s]  教学密度,纯脆敌人
   ↓
[Phase 2: 45-90s]  引入耐敌人,~20% 比例
   ↓
[Phase 3: 90-135s]  密度爬升,耐敌人 ~35%
   ↓
[Phase 4: 135-180s]  高潮,耐敌人 ~50%
   ↓
[Phase 5+: 180s+]  继续递增到必输,无上限
   ↓
[敌人触墙]
   ↓
[Game Over Overlay]  场景冻结 + "You Failed" + 按 R 重来提示
   ↓
[按 R]
   ↓
[Scene Restart]  → 回到 MainScene Phase 1
```

### 60 秒切片(玩家视角)

**0:00 - 0:30 Phase 1 前半(教学)**

屏幕上方陆续出现脆敌人(小,1 击杀),持续生成但密度低(`spawn_interval` ~ 1.5s)。玩家发飞剑,一击一个,有大量空窗时间感受单剑节奏——剑出去、远端转折、Homing 回程、入鞘、再发。偶尔池空,但敌人来得慢,池空不构成压力。

**核心体验**:熟悉操作,验证"一击一个"的反馈正确。这是玩家学习曲线的入口。

**0:30 - 0:45 Phase 1 后半(密度感建立)**

敌人密度仍是 Phase 1,但玩家已经熟悉。开始感受"持续涌入"的氛围——屏幕上同时有 3-5 个脆敌人在下落,玩家形成"扫视 → 瞄准 → 发射"的循环节奏。

**核心体验**:从"练手"过渡到"进入状态",节奏感建立。

**0:45 - 1:00 Phase 2 引入耐敌人**

第一个耐敌人(大,2 击杀)出现。玩家按习惯发剑——剑打中了,但**耐敌人没死**。这是 M3 第一个"惊讶时刻"。玩家必须:
- 视觉上记住"哪个大的还没打死"
- 飞剑回收后继续追打它,而不是去打新的脆敌人
- 这个决策过程发生在 1-2 秒内,池空期间

**核心体验**:第一次"咦没打死"的惊讶 + 单剑分配火力的取舍。这是 M2 单剑容量假设的第一次真实压力测试。

### 关键体验时刻

M3 设计追求的几个"标志性瞬间",数值调参时以这些瞬间为锚:

1. **"第一次遇到耐敌人没死"的惊讶**(Phase 2 开始时)— 设计意图明确,实施时确认"第一次出现耐敌人"的时点不要太早(玩家还没熟悉)也不要太晚(已经麻木)

2. **"池空时大敌人逼近"的紧张**(Phase 3 中期开始频繁)— M2 单剑张力假设的核心验证场景。如果玩家在这里感受到的是"麻木"而不是"紧张",数值需要重调

3. **"漏一个大敌人触墙"的失败感**(Phase 3-4 常见死法)— 失败应该感觉是"我刚才本来能打到的",而不是"不可能赢"。这是设计目标"专心就赢,分神就输"的体感锚

4. **"撑过 Phase 4 进入 Phase 5+"的成就 + 绝望**(高水平玩家)— Phase 5+ 数值继续递增到必输,玩家在这里感受到的是"我撑得久"的成就 + "迟早要死"的宿命感。这是 survivor 哲学的核心情绪

### 未来扩展提示

本节将随以下里程碑扩展:

- **M4**:加入血量系统后,"触墙瞬死" → "触墙扣血"的体验流变化。城墙血量条引入后,"撑住"的实时反馈变得连续而非二元
- **M5**:多剑武器引入后,玩家选择期 / 武器切换的体验流。生命周期可能引入"武器选择"环节
- **M6**:加入开始菜单 / 设置 / 难度选择后,启动到游戏的体验流。BootScene → MainScene 之间可能插入 MenuScene
- **M5+**:若引入音效 / 配乐,体验流的"听觉层"需要写入本节

## §4 敌人种类

### 两种敌人:差异锚点

M3 引入 **2 种**敌人,差异锚点是 **"血量 + 尺寸"正向耦合**——脆敌人小、耐敌人大,视觉直觉与玩法直觉对齐。

| 维度 | 脆敌人 | 耐敌人 |
|---|---|---|
| **血量** | 1 击杀 | 2 击杀 |
| **尺寸** | 小(~16-20 px,数值待实测) | 大(~32-40 px,数值待实测) |
| **速度** | 同速(详见 §8) | 同速 |
| **生成比例** | 阶段递增中占主导(Phase 1 100%) | 阶段递增中比例增加(Phase 2 起出现) |
| **视觉 placeholder 形状** | 简单几何形,有明确朝向(向下) | 类似形状放大 + 受击反馈 |

### 视觉 placeholder 形状要求

M2 教训 #3(memory `placeholder-shapes-need-visible-orientation`):velocity 驱动的实体必须有可见朝向,避免后续 Stage 调试时无法判断"它真的在向下移动吗"。

M3 敌人 placeholder 形状要求:

- **必须有方向感**——例如倒三角(指向下方)、楔形、箭头形状。**禁止用圆形或对称形状**
- **脆 / 耐两种敌人形状应明显可分**——不只是大小差异,**形状本身也要有差异**(例如脆 = 倒三角,耐 = 倒五边形或带头部的形状),让玩家一眼分辨而不需要"判断大小"
- **颜色编码**:配合形状,脆 / 耐使用不同颜色(具体配色待实施时定,与深蓝夜空 `#0A1428` 背景有足够对比度)
- **耐敌人受击反馈**:第一次被打中(还活着)时,**必须有视觉反馈**——例如闪烁、颜色变深、出现细微裂纹。这是"咦没打死"惊讶时刻的视觉锚点(详见 §3.5)

### Enemy interface 当前与潜在扩展

Stage 3 已就位的接口(`src/types/Enemy.ts`):

```typescript
export interface Enemy {
  takeDamage(damage: number): void;
  isAlive(): boolean;
}
```

M3 实施可能需要扩展的方法(**只在 Sword 真实需要时才加,不预设**):

- `getPosition(): { x: number; y: number }` — 若 Sword Homing 阶段需要追踪敌人(而非追踪原始转折点),则必须
- `getRadius(): number` — 若 Sword 穿透判定需要敌人碰撞圆半径,则必须
- `onDestroy(): void` — 死亡时的清理钩子(Phaser 对象销毁、池回收等)

**扩展原则**:每一个新增方法必须由具体的实施需求驱动,严禁"为了对称 / 完整 / 未来可能用到"而预先添加。M2 教训 #5 的延伸:接口设计要"基于真实需求"而非"凭空设计"。

### 击杀与穿透

M3 沿用 M2 的伤害模型:`瞬时伤害 = 当前速度 × 伤害系数`。

- **脆敌人**:被任意飞剑击中即死(瞬时伤害 > 1 HP)。这要求 `脆敌人 HP < 飞剑 V_min × 伤害系数 = 360 × 0.1 = 36`。脆敌人 HP 设为 30(预估,实测验证)
- **耐敌人**:需要 2 次击中。第一次击中扣 HP,第二次击中归零。耐敌人 HP 设为 60(预估,2 × 30,实测验证)
- **穿透**:M2 已实现"飞剑穿透敌人后按锋利度系数减速",M3 沿用。这意味着**一把飞剑可能在去程中击中多个敌人**,玩家有机会"一剑穿数敌"

### 击杀反馈

- 死亡瞬间:简单粒子飞散 + 实体消失。**M3 不做死亡动画**(留待 M5+ 美术资产阶段)
- 击杀计数:**不显示给玩家**(M3 是 placeholder 体验)。但 Debug HUD 可以加击杀计数显示,供调参用(M2 教训 #2 数据驱动 debug)

## §5 生成系统

### 生成模式:持续涌

M3 采用 **survivor 风格的持续生成**——无波间停顿,敌人按 `spawn_interval` 持续从屏幕上方涌入。生成参数由 PhaseController 按时间段切换。

### 生成位置

- **y 坐标**:屏幕上方边界外 ~20-40 px(让敌人从屏幕外"飞入"画面,避免突兀出现)
- **x 坐标**:随机 `[margin, GAME_WIDTH - margin]`,`margin` 待实测(避免敌人贴边生成导致玩家无法看清)

### 阶段递增控制(PhaseController)

由独立的 PhaseController 模块驱动,基于经过时间切换生成参数:

| Phase | 时间窗 | `spawn_interval`(预估) | 耐敌人比例 | 设计意图 |
|---|---|---|---|---|
| 1 | 0 - 45s | ~1.5s | 0% | 教学密度,纯脆敌人,玩家熟悉操作 |
| 2 | 45 - 90s | ~1.0s | ~20% | 引入耐敌人,第一次"咦没打死"惊讶 |
| 3 | 90 - 135s | ~0.7s | ~35% | 密度爬升,进入"专心就赢"核心张力区 |
| 4 | 135 - 180s | ~0.5s | ~50% | 高潮,大多数玩家在此失败 |
| 5+ | 180s+ | 继续递减 | 继续递增 | 必输区,撑得越久越好 |

所有数值标注 **(预估)**——M3 实施时按 §8 调参方法论实测确定。

### Phase 5+ 的递增策略

Phase 5+ 没有上限,需要明确递增形式:

- **`spawn_interval`**:每 30 秒线性递减 10%,直到达到下限 ~0.2s(防止 Phaser 帧率崩溃)。下限值待实施验证
- **耐敌人比例**:每 30 秒线性增加 5%,直到 ~70% 上限(留少量脆敌人维持"节奏感")
- **触发玩家必输**:数值组合在 Phase 6 末(~270s)应该让顶级玩家也无法清场。具体在哪一点必输由实测决定

### 生成节流与边界情况

- **Pause 时停止生成**:若未来加入 pause 功能,PhaseController 必须正确暂停时间累计
- **Restart 重置 Phase**:按 R 重启后,PhaseController 从 Phase 1 时间 0 重新开始
- **首发生成**:Phase 1 不要让玩家干等 1.5s 才看到第一个敌人。**第一个敌人在 0:00 - 0:02 之间生成**(具体在 0.5s 还是 1.0s 待实测,但不能等满 `spawn_interval`)

### 性能边界

M3 不做对象池 / culling 优化(M6+ 性能 milestone 处理)。预期 Phaser Graphics 在屏幕同时 20-30 个敌人下仍能保持帧率,Phase 5+ 极端情况下若帧率掉到 30 FPS 以下,在 §11 风险章节记录,推迟到 M6+ 优化。

## §6 失败与重启

### 失败触发

**触发条件**:任一敌人的 y 坐标到达城墙顶 y 坐标。

- 城墙顶 y 坐标:沿用 M1 已建立的城墙视觉位置,具体值在 `src/config.ts` 中(M3 实施时确认常量名)
- 触发判定逻辑:每帧检查所有存活敌人的 y,任一敌人 y ≥ 城墙顶 y 即触发
- 实施位置:可能在 MainScene update 循环中,或独立 TouchWallChecker 模块。实施时决定

### Game Over 状态机

触发后场景进入 "GameOver" 状态,执行以下反应:

1. **场景冻结**——所有实体停止更新:
   - 敌人停止下落
   - 飞剑停止物理推进(无论当前在去程 / Homing / 回程哪个阶段)
   - 玩家停止响应输入(A/D 移动 + 空格发射均失效)
   - 准星不再跟随鼠标(可选,实施时定)
2. **触发触墙的敌人保留在原位**——不消失,作为"罪魁祸首"的视觉锚点
3. **Game Over Overlay 显示**:
   - 屏幕中央显示文字 **"You Failed"**(英文,与游戏整体语言一致;具体字号 / 颜色待视觉调试)
   - 文字下方提示 **"按 R 重来"**
   - Overlay 不做半透明遮罩 / 模糊背景 / 动画淡入(M3 是 placeholder,M4 重做)

### Restart 流程

- **触发**:玩家按 R 键
- **执行**:`this.scene.restart()` 或等效的场景状态重置
- **结果**:
  - PhaseController 重置到 Phase 1 时间 0
  - 所有敌人清除
  - 飞剑回到入鞘态(即使刚才在去程中)
  - 玩家回到初始位置(M1 的左右中点)
  - Game Over Overlay 消失
- **不保留**:M3 不做"上局成绩"显示(留待 M4+,届时可能引入存活时间 / 击杀数统计)

### M4 重做范围

M3 placeholder 在 M4 引入血量系统后会被替换:

- **触墙瞬死 → 触墙扣血**:敌人触墙不再瞬时 game over,而是触发城墙扣血。血量归零才 game over
- **Game Over Overlay**:替换为正式 UI——可能含动画、统计(存活时间、击杀数、最高 Phase)、按钮(Restart、回到主菜单)
- **场景冻结策略**:可能改为"敌人继续推进 + 玩家不能操作"的"被吞噬"演出,或保留瞬时冻结。M4 设计时决定

## §7 BootScene 引入

### 为什么 M3 引入

CLAUDE.md 路线图明确写:**M3 — 敌人系统(从上方生成 + 向下移动 + 资源预加载,此时引入 `BootScene` 作加载器)**。M3 是引入 BootScene 的指定时机。

理由:

- **M2 之前没有真实资源**——所有视觉是 Phaser Graphics 动态画,不需要预加载
- **M3 仍可以全用 Graphics**——脆 / 耐两种敌人都简单到可以纯几何形画
- **但加载器架构需要尽早建立**——M5 接美术资产时,如果 BootScene 是 M5 才引入,会和"美术资产替换 placeholder"这两件复杂事撞在一起。先在 M3 把架构建好,M5 只需要往 BootScene 里塞资源 URL

### M3 BootScene 的最小任务

M3 实施时,BootScene 的具体内容**可能为空**——只是一个占位 Scene,完成"进入即立刻切到 MainScene"的流转。

或者,BootScene 可以**预生成敌人 texture**——把脆 / 耐两种敌人的 Graphics 一次性绘制成 Texture,存入 Phaser TextureManager,MainScene 使用 `Sprite` 而非每帧重画 `Graphics`。这是性能优化路径,**实施时根据 §5 性能边界的实测结果决定是否需要**。

### 命名约定

CLAUDE.md `## 代码风格约定` 明确:**"`MainScene` 是当前主场景。只有当需要真正预加载资源时才引入 `BootScene` 作为加载器——不要因为'看到别人都叫 BootScene'就照抄"**。

M3 引入 BootScene 是因为 CLAUDE.md 路线图本身就这么规划的(加载器架构早建立),**不是因为照抄惯例**。这一点在实施时要明确——如果 M3 BootScene 任务为空也要保留这个 Scene,是为了**建立架构占位**,M5 / M6 会用到。

### 场景流转

```
启动 (main.ts game config)
   ↓
BootScene  ← 启动 Scene
   ↓ (preload 完成 / 立即)
MainScene  ← 游戏主场景
```

`main.ts` 中 Phaser game config 的 `scene` 字段从 `[MainScene]` 改为 `[BootScene, MainScene]`,默认启动第一个。BootScene 完成后调用 `this.scene.start('MainScene')` 切换。

## §8 数值与调参方法论

### 数值表达原则

M3 设计文档中所有数值采用 **"预估区间 + 设计意图 + 实测验证条件"** 三元组表达,不写具体单值。原因:M2 教训表明,数值在 Phaser 真实运行环境下的体感与凭直觉估算的偏差可能很大(M2 V_min 从 200 调到 300 + 整体提速 20% 即为典型案例),设计阶段固定单值会让实施时被"我已经写了 1.5s"的锚定效应困住。

### 核心数值表

#### 敌人物理

| 数值 | 预估区间 | 设计意图 | 实测验证条件 |
|---|---|---|---|
| 脆敌人 HP | [25, 40] | 一击毙(任意速度的飞剑命中即死),`V_min × 伤害系数 = 36` 是死线 | Phase 1 中飞剑命中脆敌人 0 例存活 |
| 耐敌人 HP | [55, 80] | 二击毙(单击不死,二击必死),不允许出现"三击才死"的边界情况 | 全 Phase 中耐敌人被击中 0 例 3 击未死 |
| 脆敌人尺寸 | [16, 22] px | 视觉上明显小于耐敌人,且不至于在 #0A1428 背景上"看不见" | 1 米距离屏幕外可清晰识别 |
| 耐敌人尺寸 | [32, 44] px | 视觉上明显大于脆,但不至于占满 1/4 屏幕 | 同上 |
| 敌人下落速度 | [50, 120] px/s | 远低于飞剑 V_min (360),保证 INV-01 不被破坏;给玩家足够反应时间但不至于"飘" | Phase 1 中玩家失误一次后仍有 2-3s 补救窗口 |

#### 生成系统(各 Phase)

| Phase | `spawn_interval` 区间 | 耐敌人比例区间 | 设计意图 | 实测验证条件 |
|---|---|---|---|---|
| 1 | [1.3, 1.8]s | 0% | 教学密度,玩家完美操作时有 50% 空窗 | Phase 1 中完美操作玩家池空时间总和 ≥ 总时长 40% |
| 2 | [0.9, 1.2]s | [15%, 25%] | 引入耐敌人,第一次"惊讶时刻"出现在 50-55s 之间 | 玩家自述能记起"第一次没打死它"的时刻 |
| 3 | [0.6, 0.85]s | [30%, 40%] | 进入核心张力区,完美操作下 0-1 漏 | 中等水平玩家在 Phase 3 内失败率 30-50% |
| 4 | [0.4, 0.6]s | [45%, 55%] | 高潮,大多数玩家失败 | 顶级玩家通过率 < 30%,中等玩家通过率 < 5% |
| 5+ | 每 30s 递减 10%,下限 0.2s | 每 30s 递增 5%,上限 70% | 必输区,撑得越久越好 | Phase 6 末(~270s)顶级玩家也无法清场 |

#### 视觉与反馈

| 数值 | 预估值 | 设计意图 | 实测验证条件 |
|---|---|---|---|
| 耐敌人受击反馈持续时间 | 80-150ms | 短促但可见,不延误玩家判断 | 玩家自述能感知到"打中了但没死" |
| Game Over Overlay 文字字号 | 待视觉调试 | 屏幕中央显眼,不过分占屏 | 30cm 距离一眼可读 |
| 敌人生成 y 坐标 | -20 ~ -40(屏幕外) | "飞入"画面而非"出现"在画面内 | 玩家不会看到敌人凭空出现的瞬间 |

### 调参方法论

#### 阶段验证策略

不要一次性调全 3 分钟。按阶段验证:

1. **Phase 1 单独验证**:把 PhaseController 锁在 Phase 1 数值无限循环,调脆敌人 HP / 速度 / `spawn_interval`,直到"完美操作 0 漏 + 适度空窗"达成
2. **Phase 1-2 过渡验证**:解锁到 Phase 2,重点调耐敌人 HP / 受击反馈 / Phase 2 `spawn_interval`,直到"第一次惊讶"自然发生
3. **Phase 3-4 压力验证**:解锁到 Phase 4,重点调密度递增曲线,目标是"中等玩家在 Phase 3-4 间失败"
4. **Phase 5+ 必输验证**:解锁全程,确认 Phase 5+ 数值真的能让顶级玩家失败,且不在某个数值组合下"卡住"(例如敌人持续涌但都打得过)
5. **全 3 分钟体感验证**:完整跑 3 分钟,检查节奏曲线是否流畅,有没有"突然变难"或"突然变水"的断层

#### 数据驱动 debug(M2 教训 #2)

调参期间 Debug HUD 必须显示:

- 当前 Phase 编号 + Phase 内时间
- 当前 `spawn_interval` + 耐敌人比例(实时,验证 PhaseController 工作正确)
- 屏幕同时存在的敌人数 + 脆 / 耐分别计数
- 累计击杀数(脆 + 耐)+ 累计漏过数(虽然 M3 不显示给玩家,debug 时显示)
- 玩家飞剑状态(in-pool / out-flying / homing)
- 玩家本次存活时间

这些数据用于在"觉得不对"时立刻量化"哪里不对"——是 `spawn_interval` 偏快?是耐敌人比例偏高?是玩家自己 miss 太多?数据驱动反驳 / 验证用户体感(包括设计者自己的体感)。

#### 接受"完成 > 完美"(M2 教训 #1)

M3 数值目标是 **"能玩出张力的最小可调参版本"**,**不追求完美平衡**。具体:

- 如果调到"Phase 3 中等玩家失败率在 25% 或 55%",都可接受;追求"恰好 40%"是浪费 M3 时间
- 如果 Phase 4 顶级玩家通过率"在 25% 或 35%",都可接受;追求"恰好 30%"是浪费
- 如果某个数值"看起来不优雅"但体感不出问题,接受,标 TODO,M4 / M5 调

**M3 数值的合格线**:
- 三个"关键体验时刻"(§3.5)出现 — 惊讶 / 紧张 / 失败感
- 完整 3 分钟内没有"麻木期"超过 30s
- Phase 间切换没有"断层突兀感"

达到合格线即停止调参。完美平衡是 M5+ 的事。

#### 预计调参成本

参考 M2 调参轮次,预计 M3 调参成本为 **M2 的 1.5-2x**。原因:

- 旋钮数量:M2 调 ~6 个数值(V_max/min, D_max, 锋利度, 伤害系数, 角速度),M3 调 ~15 个(各 Phase 的 spawn_interval × 5 + 耐比例 × 5 + 敌人 HP × 2 + 尺寸 × 2 + 下落速度 × 1)
- 旋钮间耦合:M2 数值大多独立调,M3 阶段间的连续性带来耦合(Phase 2 调动会影响 Phase 3 体感)
- 验证场景多:M2 验证"飞剑物理本身",M3 验证"完整 3 分钟体验",单次完整测试就要 3 分钟

PM 心理预期:M3 数值调参可能占用 M3 总实施时间的 30-40%。

## §10 决策追溯

本节归档 M3 设计过程中所有"为什么不选 X"的反对方案与理由。今天(2026-05-25)产出对话中讨论的所有分叉点都在此记录,服务未来里程碑回看与可能的设计反复。

### §10.1 闭环 vs 不闭环

**最终选**:(a) M3 闭环 — 触墙 = placeholder game over

**反对方案**:
- **(b) M3 不闭环**:敌人触墙就消失,无失败状态,M3 = "敌人系统",M4 = "游戏循环 + 输赢"
  - 反对理由:M3 完成后游戏仍是"不会输的演示",没有 demo 价值;PM 希望 M3 是独立可演示里程碑
- **(c) M3 半闭环**:敌人触墙不消失也不 game over,堆积在墙上,玩家自己看着办
  - 反对理由:"没有输就没有真正的张力",漏 10 个还是 100 个玩家感受可能一样麻木;且不省 M3 范围(仍需视觉处理堆积)

### §10.2 张力定位

**最终选**:(2) 紧绷张力 — "专心就赢,分神就输"

**反对方案**:
- **(1) 训练场张力**:很难输,失败是惩罚
  - 反对理由:太松,池空感觉不到,无法验证 M2 单剑张力假设
- **(3) 必输张力**:roguelike survivor 那种持续涌
  - 反对理由:太紧,玩家可能根本顾不上感受单剑节奏

注:Phase 5+ 实际就是 (3) 必输张力,但作为"3 分钟之后"的边界形态而非主体,不冲突。

### §10.3 敌人耐打度

**最终选**:(γ) 混合 — 脆 + 耐两种

**反对方案**:
- **(α) 全脆 / 一击杀**:只一种敌人
  - 反对理由:玩家 30 秒后熟练就麻木,"3 分钟张力容量"撑不住;过分简化掉了"分配火力"的取舍
- **(β) 全耐 / 多剑磨**:只一种敌人,需要 2-3 剑才杀
  - 反对理由:单剑容量瓶颈被血量盖住,失去了"节奏感";数值复杂度高(HP / 衰减 / 锋利度 / 穿透耦合)

### §10.4 敌人差异维度耦合方向

**最终选**:血量 + 尺寸正向耦合 — 脆小、耐大

**反对方案**:
- **血量 + 速度反向耦合**:耐打的反而快,脆的反而慢
  - 反对理由:视觉传达难(大的看起来不该慢),反直觉留 M5+ 多剑武器引入后再玩
- **只血量不耦合**:两种敌人看起来一样,只是 HP 不同
  - 反对理由:玩家视觉上分不清,会感觉乱
- **血量 + 颜色单独耦合**:大小一致,只颜色区分
  - 反对理由:颜色区分弱于尺寸区分,玩家视野扫描时更容易漏判

### §10.5 难度递增形式

**最终选**:持续涌 + 阶段递增(基于时间切换 spawn 参数)

**反对方案**:
- **(i) 时间线性递增**:从 0 到 3:00 平滑增加
  - 反对理由:玩家感受不到节点,温水煮青蛙
- **(ii) 波次 + 停顿**:一波出 N 个,清空后停顿,再下一波
  - 一度选中,后改:停顿提供节奏缓冲与"撑过一波"的成就感,但 PM 决定要"持续涌"的体验
- **(iii) 阶梯式**:每 60s 阶跃式提升,中间平稳
  - 反对理由:阶跃感可能太突兀;且"持续涌阶段递增"已经是连续阶梯的更平滑版本

### §10.6 结局形态

**最终选**:(B3) Phase 5+ 必输,只有 You Failed,无 You Win

**反对方案**:
- **(B1) 3:00 通关**:撑到 3:00 显示 You Win
  - 反对理由:不符合 survivor 哲学,需要做 You Win UI(M3 范围扩);"3 分钟"的初心是张力容量,不是通关时长
- **(B2) 3:00 后停止递增**:Phase 4 数值永久持续
  - 反对理由:没有"通关"也没有"必输",顶级玩家可能无限玩下去,缺乏结局感

### §10.7 屏幕敌人堆积处理

**最终选**:(A1) 不处理,触墙做兜底

**反对方案**:
- **(A2) 屏幕敌人数硬上限**:超过 N 个停止 spawn
  - 反对理由:改变设计语义,"密度递增"被打断
- **(A3) 性能优化层**:对象池 + culling
  - 反对理由:M3 范围外,M6+ 性能 milestone 处理

### §10.8 文档形态

**最终选**:M2 同款双文档结构 — 今天先产出 design 文档,implementation guide 临近实施时产出

**反对方案**:
- **单文档**:design + implementation guide 合并
  - 反对理由:M2 的双文档节奏证明有效(design 稳定,guide 接近实施时写更准);M3 沿用
- **三文档**:design + guide + 数值表独立
  - 未提出,但理论可能。反对理由:M3 范围不至于到需要数值表独立的程度

### §10.9 §4-§5 章节内是否带 mini 决策追溯

**最终选**:章节内不带,全归 §10 集中

**反对方案**:
- **章节内带 mini 追溯**:读者读机制时上下文齐全
  - 反对理由:与 §10 重复;选择信息密度更高的章节正文 + 集中追溯归档

### §10.10 BG_COLOR 债是否写入 M3 设计文档 §2

**最终选**:不写,只在 memory `m2-bg-color-debt.md` 中维护

## §11 已知风险与未来工作

### M3 实施期已知风险

#### 风险 1:3 分钟从头紧绷的心流疲劳

**描述**:M3 取消波间停顿后,玩家从 0:00 起就处于"持续涌入"压力下,3 分钟匀速紧绷可能在 Phase 3-4 引发心流疲劳——玩家不是"被打败",而是"累了不想玩了"。

**触发条件未知**:可能在 Phase 3 中段(约 2:00)显现,也可能在数值调到位后自然消失(若节奏曲线足够好玩家会沉浸)。

**应对**:实测中若 Phase 3 玩家自述"累",考虑 Phase 间引入极短的密度低谷(2-3s 内 `spawn_interval` 短暂回升)作为"假停顿",而非完全停顿。M3 范围内不预先设计此机制,实测显现再加。

#### 风险 2:Phaser Graphics 多实体性能

**描述**:M3 不做对象池 / culling 优化,Phase 5+ 极端情况下屏幕同时存在敌人数可能达到 30-50 个,Phaser Graphics 动态绘制成本上升。

**触发条件**:Phase 5+ 末段(~270s 之后)若顶级玩家仍未失败,实体数持续累积。

**应对**:
- M3 实施时 Debug HUD 必须显示当前帧率
- 若 30 FPS 以下持续超过 5s,记录到 M6+ 优化任务,**不在 M3 范围内修**
- 若帧率崩溃发生在 Phase 5+ 之前(即正常游戏过程中),则是真问题,M3 内必须处理(可能用 BootScene 预生成 texture 替代每帧 Graphics)

#### 风险 3:数值调参轮次拖慢 M3 周期

**描述**:§8 预估 M3 调参成本为 M2 的 1.5-2x,占总实施时间 30-40%。若实际调参遇到强耦合(Phase 2 调动连环影响 Phase 3-4),周期可能进一步拉长。

**应对**:
- 接受"完成 > 完美"(M2 教训 #1),数值合格线(§8)达到即停止
- 阶段验证策略(§8)让单次测试聚焦,避免每次都跑全 3 分钟
- 若 M3 第 3 周仍未通过 Phase 3 数值验证,触发设计反思——可能是 framing 本身有问题(例如"专心就赢"在当前 V_max/V_min 下数学上不成立),需要回到设计文档调整 §3

#### 风险 4:单剑周期估算未交叉验证

**描述**:§3 中"单剑完整周期 ≈ 1.5-2.3s"是估算值,与 M2 设计文档 §8 的实测数值未交叉确认。若实测周期显著短于或长于估算,则 §5 各 Phase `spawn_interval` 预估区间需要全表重调。

**应对**:M3 实施第一步是确认 M2 实测单剑周期,作为所有生成数值的锚点。
