# 项目级设计不变量

> **用途**: 跨里程碑使用的强约束清单. 任何后续里程碑实施都不得违反.
> **维护**: 新里程碑设计文档定稿后, 在本文档追加新不变量; 不在实施阶段临时引入.

---

## §1 什么是设计不变量

**不变量是强约束, 不是临时兜底**.

- 不变量来自里程碑设计文档定稿后的提炼 — 是设计阶段经过对齐的"游戏说不上为什么但就是这样"的底层假设
- 任何后续迭代必须遵守; 违反不变量需要在新里程碑设计文档中**显式声明并讨论后果**, 不能在实施阶段悄悄绕过
- 不变量不是"建议", 是"边界"; 一旦突破, 涉及的题材、手感、机制平衡会同时崩塌, 修复成本远高于初始遵守
- 写代码、加 buff、调数值、设计技能时, 凡是触及不变量的设计点, 都必须先验证是否会突破

每条不变量都应能回答三个问题: **是什么规则 / 实现层面要怎么做 / 为什么不能违反**.

## §2 当前不变量

### INV-01 飞剑速度恒 > 角色速度

**来源**: M2 设计文档 §1.2 / §3.7

**规则**: 飞剑的任何状态下的瞬时速度, 必须大于角色在任何状态下的瞬时速度.

**实现含义**:

- 角色移动速度上限设定时, 必须保证 `playerMaxSpeed < V_min` (M2 阶段 V_min = 200 px/s)
- "任何状态"包括所有 buff / 加速技能 / 装备效果叠加后的状态
- 飞剑速度下限 (M2 = 100 px/s) 是物理兜底, 防止剑卡死, **不**是给角色突破用的口子
- M5+ 加角色加速类技能时, 必须同步检查不会突破此约束 (或同步抬升 V_min)

**为什么是不变量**: 飞剑永远是主动的, 角色不能靠跑赢飞剑来"丢弃"它. 一旦角色能跑赢飞剑, "御剑"的题材语义崩塌 — 剑变成可被丢弃的弹幕, 玩家会发现"快跑就没事". 这条不变量是单剑模型与回程追踪能成立的物理底座.

---

### INV-02 飞剑数量 = 资源池容量

**来源**: M2 设计文档 §1.2 / §2.2

**规则**: 飞剑系统始终是"资源池"语义, 不是"武器堆叠"语义. 同时存在的飞剑数量 ≤ 池容量上限, 等于"已派出 + 池中可用".

**实现含义**:

- 飞剑容量是角色属性, 不是关卡掉落 / 装备叠加 / 时间累积
- 池空状态 (所有剑都在外飞行) 是被有意设计的张力时刻, **不能**通过自动补剑 / CD 重置 / 缩短往返时间等方式消除
- M5+ 加多剑成长时, 仍然走"角色等级 → 池容量 +1"的路径; 不引入"任务奖励 +1 剑"、"装备 +2 剑容量"这类堆叠机制
- 飞剑技能 (剑诀) 消耗的也是池中飞剑, 而不是独立的法力/能量条

**为什么是不变量**: 这条约束让单剑模型 (M2) 和多剑成长 (M5+) 共享同一套语义和代码骨架, 避免出现"M2 是单剑、M5 是堆叠武器"的设计分裂. 同时池空被有意保留为张力时刻 — 一旦池容量可以无限堆叠 / 池空可以瞬间补满, 玩家就再也不需要走位避敌, 单剑模型设计的核心张力消失.

---

### INV-03 机制成长 = 修为成长

**来源**: M2 设计文档 §1.2 / §5

**规则**: 飞剑系统的能力解锁 (容量增加 / 召回技能 / 多种剑 / 剑诀 / 御剑能力倍率) 通过**角色等级 / 修为系统**提升, 而**不**通过装备掉落 / 科技树 / 任务奖励等外部物品获得.

**实现含义**:

- 角色御剑能力 (初速倍率 / 最大距离倍率 / 锋利度倍率 / 伤害系数倍率 / 回程速度倍率) 由 `PlayerStats` 持有, 来源是角色等级 / 技能点 / 修为
- 多种飞剑武器 (M5+) 的解锁也走"修为达到 X → 可御 Y 剑", 不走"打开宝箱抽到 Y 剑"
- **属性是数据, 不是常量** — 飞剑实际属性必须是 `基础属性 × 御剑能力倍率`, 任何写死的 `SWORD_SPEED = 1000` 都违反此不变量
- M2 阶段御剑能力倍率全 = 1.0, 但 `PlayerStats` 框架必须就位, 否则 M5+ 时需要大幅重构

**为什么是不变量**: 题材是修真, 御剑能力即修为. 一旦引入装备/科技树, 题材一致性会崩塌 — 玩家会问"为什么我的剑变快是因为捡到一个戒指". 同时这条约束保证了所有飞剑成长都汇聚到一个数据通道 (`PlayerStats`), 避免后期出现"装备加成、buff 加成、技能加成、装备技能联动加成"的多源叠加噩梦.

## §3 追加新不变量的规则

**何时追加**:

- 新里程碑设计文档定稿后, 设计中浮现的"任何后续迭代都不得违反"的强约束, 整理成新条目
- **不在实施阶段临时引入** — 实施阶段发现新约束, 应回到设计文档讨论, 而非直接写进本文档

**编号**: 顺序递增 INV-XX, 不重用已删除的编号.

**必填字段** (每条不变量都需要):

- **来源**: 引用里程碑设计文档的具体章节
- **规则**: 一句话说清"什么不能违反"
- **实现含义**: 代码 / 数值 / 系统层面如何遵守
- **为什么是不变量**: 一旦违反会发生什么 — 这是判断边界情况时的依据

**修改 / 废弃**: 不变量一经确立不轻易改动. 如需修改或废弃, 在新里程碑设计文档中显式说明动机与影响范围, 本文档保留旧条目并标注"已于 MX 废弃, 见 [里程碑设计文档链接]", 不要直接删除.
